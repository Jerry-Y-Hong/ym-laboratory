## 1. YM-LAB Engineering Principles (YM-LAB 엔지니어링 4대 원칙)

YM-LAB PROJECT의 모든 설계와 소스코드는 다음 4대 엔지니어링 원칙을 지침으로 삼는다.

```
┌─────────────────────────────────────────────────────────────┐
│ 1. Integrity (무결성): 데이터 및 지식 관계의 왜곡 방지           │
├─────────────────────────────────────────────────────────────┤
│ 2. Simplicity (단순성): 복잡한 외부 분산 기술 도입 차단            │
├─────────────────────────────────────────────────────────────┤
│ 3. Stability (안정성): 벤더나 모델이 바뀌어도 유지되는 구조        │
├─────────────────────────────────────────────────────────────┤
│ 4. Traceability (추적성): 요구사항-설계-코드-테스트 간 1:1 매핑    │
└─────────────────────────────────────────────────────────────┘
```

### 1.1 Integrity (지식 무결성)
- 모든 식품 분류 및 가공, 영양소 매핑은 자의적으로 판단하지 않으며, `300_KNOWLEDGE_ENGINE` 원천 데이터베이스를 기준으로 그라운딩(Grounding)하여 오차율 0%를 유지한다.
- catalog.db와 로컬 JSON 데이터셋의 해시 무결성을 체크하여, 소스 유출이나 원천 데이터의 변조를 수시 방지한다.

### 1.2 Simplicity (로컬 단순성)
- 프로젝트는 경량 로컬 개발을 지향한다. Redis, Kafka, Elasticsearch, AWS Cloud-Native 전용 자원 등의 고비용 분산 인프라를 무단 도입하지 않는다.
- 데이터 영속성 제어 시 SQLite와 JSON 파일 포맷 등 파일 I/O를 활용해 단순하고 명확하게 구현한다.

### 1.3 Stability (프레임워크 안정성)
- 모델이나 벤더사가 대체되더라도 프레임워크는 깨지지 않아야 한다:
  > *The operating framework must remain stable even if AI models, tools or vendors are replaced in the future.*
- 코딩 작성 시 파이썬 표준 라이브러리 및 하위 호환성이 보장된 기본 패키지를 주로 활용하여, 런타임 환경의 버전 호환 오류 가능성을 사전에 최소화한다.

### 1.4 Traceability (추적성)
- 모든 기능은 요구사항 정의서(`01_requirements.md`)의 특정 ID에서 출발하여 시스템 설계서, 구현 코드 주석, 검증 스크립트(`verify_*.py`)와 완전히 유기적으로 매핑되어야 한다.

## 2. Core Development Standards

- **로컬 검증**: 모든 변경 코드는 Git 커밋 전에 로컬에서 검증 스크립트를 수동 구동하여 정합성을 확인한다.
- **아키텍처 동결 준수**: `PROJECT_STATUS.md` 상 🔒 Closed 상태로 동결된 Phase 01~10 산출물 파일은 절대 덮어쓰거나 수정하지 않는다.
- **수동 최종 승인 (HITL)**: 코드 변경 시 인간 프로젝트 리더(Human Project Lead)의 검토와 승인을 거쳐 병합하는 것을 표준으로 삼는다.
- **예외 처리와 로깅**: 예외는 최상위 런타임까지 전파되어 프로세스가 죽지 않도록 방어 설계하며, 파이썬 표준 로깅(Python Logging Standard) 방식을 사용하여 기록한다.
- **환경 분리**: 로컬 설정은 소스코드 내에 하드코딩하지 않고 환경별 설정(Configuration) 및 세팅(Settings) 수칙을 적용해 외부에서 주입한다.
