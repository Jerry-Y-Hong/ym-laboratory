# 📘 MFCO DESIGN BLUEPRINT v1.1 & CORE DESIGN PRINCIPLES v1.0

## A. 철학 (Philosophy)
MFCO는 의학 분류 체계를 구축하는 프로젝트가 아니다. 
MFCO는 사용자가 경험하는 상태(State)를 출발점으로 하여, 상태를 분석하고, 상태를 유발하는 원인(Cause)을 추론하며, 원인 해결에 필요한 역할(Role)을 도출하고, 역할을 수행하는 기능(Function)을 연결한 뒤, 기능을 제공하는 식재료(Ingredient)를 선택하고, 최종적으로 레시피(Recipe)를 설계하는 **의미 기반 추론 엔진(Semantic Inference Engine)** 이다.

### MFCO Core Flow
```
STATE ──> CAUSE ──> ROLE ──> FUNCTION ──> INGREDIENT ──> RECIPE
```

### MFCO의 핵심 철학
* 질병(Disease) 중심이 아니다.
* 처방(Prescription) 중심이 아니다.
* 약재(Herb) 중심이 아니다.
* 식재료(Food) 중심도 아니다.
* MFCO는 **"사용자가 현재 어떤 상태를 경험하고 있는가?"**를 가장 중요한 출발점으로 삼는다.

## B. Ontology Rule
모든 추론 노드는 의미 기반으로 관리되며, 아래 계층 관계를 고수한다.

---

## C. NODE 판정 규칙 (Node Determination Rules)
새로운 데이터가 들어오면 반드시 아래 순서로 판단하여 계층을 분류한다.

* **STEP 1: 사용자 표현인가?**
  * **YES** ➡️ `STATE` (예: 부종, 냉증, 피로, 손발이 차다)
  * **NO** ➡️ `STEP 2`
* **STEP 2: 원인 설명인가?**
  * **YES** ➡️ `CAUSE` (예: 습담, 기허, 혈허)
  * **NO** ➡️ `STEP 3`
* **STEP 3: 개입 방향인가?**
  * **YES** ➡️ `ROLE` (예: 습제거, 보기, 활혈)
  * **NO** ➡️ `STEP 4`
* **STEP 4: 생리 작용인가?**
  * **YES** ➡️ `FUNCTION` (예: 이수, 건비, 보기, 활혈)
  * **NO** ➡️ `재검토`

---

## D. FUNCTION 검증 규칙 (Function Validation Rules)
새로운 `FUNCTION`을 등록하기 전 반드시 아래 규칙들을 통과해야 한다.

* **RULE F-01 (상태가 아닌가?)**: `부종` ❌, `냉증` ❌와 같은 상태 지표는 제외.
* **RULE F-02 (원인이 아닌가?)**: `기허` ❌, `혈허` ❌, `습담` ❌와 같은 변증 원인은 제외.
* **RULE F-03 (생리 작용인가?)**: `이수` ✔, `건비` ✔, `보기` ✔, `활혈` ✔과 같은 직접적인 생리적 액션만 포함.
* **RULE F-04 (독립 기능인가?)**: 더 작은 기능으로 분해 가능한 복합적 용어(예: `혈액순환개선`)는 하위 독립 기능(예: `활혈`, `미세순환촉진`, `혈관확장`)으로 쪼개고, 상위 시스템 기능으로 이동시킨다.

---

## E. FUNCTION 구축 원칙
* **절대로** 증상 사전이 아니다.
* **절대로** 변증 사전도 아니다.
* **절대로** 효능 마케팅 문구도 아니다.
* `FUNCTION MASTER`는 오직 **인체가 수행하는 생리 기능**만 저장한다.

---

## F. 추론 엔진 규칙
* **직접 연결 절대 금지**: `STATE`에서 `INGREDIENT`로 직접 화살표가 이어지지 않는다.
  * *잘못된 예*: `부종` ➡️ `옥수수수염` ❌
  * *올바른 예*: `부종 (STATE)` ➡️ `습담 (CAUSE)` ➡️ `이수 (FUNCTION)` ➡️ `옥수수수염 (INGREDIENT)` ✔

---

## G. MFCO 검증 질문
새로운 노드를 추가할 때 항상 아래 5가지 질문을 거친다.
1. 이것은 상태(State)인가?
2. 이것은 원인(Cause)인가?
3. 이것은 역할(Role)인가?
4. 이것은 기능(Function)인가?
5. 어느 계층에 속하는가?

---

## H. 최종 정의 (Core Mission Statement)
> **"MFCO는 사용자 상태를 의미 기반으로 해석하여, 원인·기능·식재료·레시피로 연결하는 다층 추론 엔진이다."**

## 📘 MFCO CORE DESIGN PRINCIPLES v1.0

* **P-01 의미 중심 설계 (Meaning First)**: 단어가 아니라 의미를 기준으로 판단한다. (동의어 ➡️ 병합, 이음동의어 ➡️ 분리)
* **P-02 병합할 것은 병합 (Merge)**: 동일한 기능은 하나의 노드로 통합한다. (예: `이수` & `이뇨` ➡️ 동일 FUNCTION)
* **P-03 분리할 것은 분리 (Split)**: 서로 다른 의미는 분리한다. (예: `청열` & `해독` ➡️ 각각 개별 기능으로 분리)
* **P-04 계층 구조 유지 (Hierarchy)**: 모든 노드는 상위/하위 계층 구조를 가진다.
* **P-05 원문 필수 보존 (Source Preservation)**: 원문 용어와 현대 표준어를 동시에 보관한다. (원문 용어를 버리면 전통 문헌과의 추적이 불가능해진다.)
* **P-06 현대어 표준기능 병행 사용**: 일반 사용자/AI 추론은 현대어 기준, 전통 지식 추론은 원문 기준으로 둘 다 지원한다.
* **P-07 Translation Layer 유지**: 원문과 현대 표준 기능 간의 양방향 매핑을 보장한다. (예: `活血` ↔ `혈액순환촉진`)
* **P-08 역추적 가능성 보장 (Traceability)**: 레시피 추천부터 증상까지 역방향 설명이 가능해야 한다.
* **P-09 추론 체인 고정**: `STATE ➡️ CAUSE ➡️ ROLE ➡️ FUNCTION ➡️ INGREDIENT ➡️ RECIPE` 체인을 고수한다.
* **P-10 원문 데이터 불변 원칙 (Immutable Source Rule)**: RAW 원천 데이터는 절대 수정하지 않으며, 정제 파이프라인(`RAW ➡️ NORMALIZED ➡️ MFCO NODE`)을 통해서만 가공한다.
* **P-11 모든 노드는 ID 기반 관리**: 노드 간의 매핑은 이름이 아닌 고유 ID(예: SF001, RC001)를 기반으로 작동한다.
* **P-12 설명 가능한 AI (Explainable AI)**: 모든 추천 결과는 논리적 근거를 설명할 수 있어야 한다.
* **P-13 ROLE ↔ FUNCTION 다대다(M:N) 허용**: 역할과 기능은 1:N이 아닌 다대다 관계를 가진다. (예: '건비' 기능은 습제거, 기력회복, 영양흡수개선 등 여러 역할에 기여할 수 있고, '습제거' 역할은 이수, 건비, 조습 등 여러 기능을 취할 수 있다.)
