## 1. Executive Summary & Repository Vision

The **Knowledge Repository Standard** specifies the enterprise data organization, schema models, directory structures, and metadata standards governing all knowledge assets stored within the AI Content Production Platform (ACPP). 

The primary architectural objective is to establish an immutable **Single Source of Truth (SSOT)** for enterprise domain knowledge. The repository enforces strict separation between **Core Domain Knowledge** (channel-agnostic, factual, structured) and **Generated Channel Artifacts** (format-specific, rendered, published).

This standard ensures complete domain portability across current and future enterprise domains, including **Kimchi Research** (`01_PHASE1_KIMCHI`), **Smart Farm Automation**, **MFCO Knowledge Base**, **Patent Intelligence**, and **AI Enterprise Core Specifications**.

```
┌────────────────────────────────────────────────────────────────────────┐
│                   ACPP KNOWLEDGE REPOSITORY TOPOLOGY                   │
├────────────────────────────────────────────────────────────────────────┤
│ raw/             Ingested un-structured research & source data         │
├────────────────────────────────────────────────────────────────────────┤
│ structured/      Normalized, Q-Code indexed SSOT Knowledge Assets      │
├────────────────────────────────────────────────────────────────────────┤
│ published/       Generated multi-channel content artifacts (Blog, PDF) │
├────────────────────────────────────────────────────────────────────────┤
│ index/           Global Q-Code taxonomy, vector embeddings & graph maps│
└────────────────────────────────────────────────────────────────────────┘
```

## 2. Directory Layout Standard

Every ACPP Knowledge Repository instance strictly complies with the standardized 4-tier directory taxonomy:

```
<domain_repository_root>/
├── raw/
│   ├── web_scrapes/
│   ├── literature_pdfs/
│   └── database_dumps/
├── structured/
│   ├── assets/
│   │   ├── KA-KIMCHI-001.md
│   │   └── KA-KIMCHI-002.json
│   └── schemas/
│       └── asset_schema_v3.json
├── published/
│   ├── blog/
│   ├── website/
│   ├── pdf_books/
│   ├── slides/
│   └── newsletters/
└── index/
    ├── qcode_registry.json
    ├── vector_index/
    └── knowledge_graph.json
```

### Subdirectory Specifications

1. **`raw/` (Raw Research Vault)**
   - Houses unmodified, source data collected by the Research Agent.
   - Files are append-only and cryptographically hashed (`SHA-256`) to ensure research provenance.
2. **`structured/` (Single Source of Truth Repository)**
   - Contains normalized, verified Knowledge Assets.
   - Format: Dual representation—human-readable YAML-frontmatter Markdown (`.md`) and validated JSON (`.json`).
3. **`published/` (Channel Artifacts Vault)**
   - Stores channel-specific rendered outputs generated by Writing, SEO, and Publishing Agents.
   - Organized by output channel and publication timestamp.
4. **`index/` (Enterprise Taxonomies & Vector Indices)**
   - Holds global Q-Code registries, domain taxonomies, and vector embedding indices for semantic similarity search.

## 3. Metadata Standard & Metadata Schema

Every Knowledge Asset within `structured/` must incorporate the mandatory **ADF v3.1 Metadata Standard** block:

| Field Name | Type | Description | Mandatory |
|---|---|---|---|
| `asset_id` | String | Unique asset key (e.g. `KA-KIMCHI-2026-001`) | Yes |
| `qcode` | String | Enterprise Q-Code identifier (e.g. `Q-KIM-FERM-042`) | Yes |
| `domain` | String | Enterprise domain code (`KIMCHI`, `SMART_FARM`, `MFCO`, `PATENT`, `ADF_DOCS`) | Yes |
| `title` | String | Human-readable canonical title of the knowledge node | Yes |
| `version` | String | Semantic version string (`v1.0.0`) | Yes |
| `author_agent` | String | Agent role code responsible for asset creation (`ACPP-AG-02`) | Yes |
| `verification_score` | Float | Fact verification rating from `0.00` to `1.00` | Yes |
| `security_level` | String | Access control classification (`PUBLIC`, `INTERNAL`, `RESTRICTED`, `CONFIDENTIAL`) | Yes |
| `sources` | Array | Reference list linking back to `raw/` files or external URIs | Yes |
| `last_audited_utc` | String | ISO 8601 timestamp of last human or agent audit | Yes |

## 4. Declarative Knowledge Asset Schema Specification

Knowledge Assets are defined via a formal declarative JSON Schema:

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "ACPP_KnowledgeAsset_v3.1",
  "type": "object",
  "required": [
    "asset_id",
    "qcode",
    "domain",
    "title",
    "version",
    "verification_score",
    "sources",
    "claims",
    "entities"
  ],
  "properties": {
    "asset_id": { "type": "string", "pattern": "^KA-[A-Z0-9_-]+$" },
    "qcode": { "type": "string", "pattern": "^Q-[A-Z0-9_-]+$" },
    "domain": { "type": "string" },
    "title": { "type": "string" },
    "version": { "type": "string", "default": "v1.0.0" },
    "verification_score": { "type": "number", "minimum": 0.0, "maximum": 1.0 },
    "sources": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["source_id", "uri", "credibility"],
        "properties": {
          "source_id": { "type": "string" },
          "uri": { "type": "string" },
          "credibility": { "type": "number" }
        }
      }
    },
    "claims": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["claim_text", "evidence_source_id"],
        "properties": {
          "claim_text": { "type": "string" },
          "evidence_source_id": { "type": "string" }
        }
      }
    },
    "entities": {
      "type": "array",
      "items": { "type": "string" }
    }
  }
}
```

## 5. Standard Markdown Frontmatter Representation

Below is the standard dual-format Markdown Knowledge Asset template used in `structured/`:

```markdown
---
asset_id: "KA-KIMCHI-2026-088"
qcode: "Q-KIM-FERM-008"
domain: "KIMCHI"
title: "Temperature Dynamics and Microorganism Kinetics in Baechu Kimchi Aging"
version: "v1.0.0"
author_agent: "ACPP-AG-02"
verification_score: 0.98
security_level: "PUBLIC"
last_audited_utc: "2026-07-23T12:00:00Z"
sources:
  - source_id: "SRC-RAW-001"
    uri: "file:///01_PHASE1_KIMCHI/05_FERMENTATION_MASTER/fermentation_specs.json"
    credibility: 1.00
entities:
  - "Leuconostoc mesenteroides"
  - "Lactobacillus plantarum"
  - "Lactic Acid"
---

# Asset Summary
Baechu Kimchi fermentation kinetics depend heavily on ambient temperature. Storage at 4°C slows down *Lactobacillus plantarum* proliferation while encouraging *Leuconostoc mesenteroides*, producing optimal carbonation and crisp texture.

## Key Claims & Empirical Evidence
1. **Low-Temperature Aging (4°C)**: Extends optimal taste window up to 90 days.
   - *Evidence Source*: [SRC-RAW-001]
2. **High-Temperature Fermentation (15°C+)**: Causes rapid acidification within 48 hours.
   - *Evidence Source*: [SRC-RAW-001]
```

## 6. Enterprise Domain Portability Model

The repository standard is completely domain-agnostic. Onboarding a new knowledge domain requires zero code changes to the underlying platform engine; it only requires registering a domain configuration file in `index/domain_config.json`:

```json
{
  "registered_domains": [
    {
      "domain_code": "KIMCHI",
      "domain_name": "Kimchi Heritage & Science Base",
      "qcode_prefix": "Q-KIM-",
      "schema_extension": "schemas/kimchi_ext.json"
```
